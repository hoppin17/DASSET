# 온체인 데이터 수집 메모

## 분석 대상

트랜잭션(transaction)에는 코인 전송 외에도 투표, 컨트랙트 실행 등 다양한 활동이 포함됩니다. 이 프로젝트에서는 코인의 이동과 송신·수신 주소를 확인할 수 있는 전송(transfer) 내역을 주요 분석 대상으로 삼았습니다.

분석한 자산은 다음과 같이 구분했습니다.

- 독립된 체인을 가진 자산: Bitcoin, Ethereum, Ripple, Solana 등
- 기존 체인의 컨트랙트로 발행된 토큰: Cobak Token, SOMESING, Pepe 등

독립 체인을 가진 자산은 해당 체인의 트랜잭션에서 전송 내역을 확인했습니다. 별도 체인이 없는 토큰은 토큰이 발행된 체인에 기록된 컨트랙트 기반 전송 내역을 사용했습니다.

## 데이터 수집 여건

여러 체인의 과거 데이터를 일괄 제공하는 서비스는 많지 않았습니다. 직접 노드를 운영하려면 높은 사양의 장비와 운영 비용이 필요해 프로젝트 여건상 선택하기 어려웠습니다. 각 체인이 제공하는 API도 특정 지갑이나 개별 거래 조회에 초점을 둔 경우가 많아, 전체 기간의 전송 내역을 수집하는 데 한계가 있었습니다.

## 실제 사용한 방법

### Bitquery

Bitquery는 여러 체인의 데이터를 GraphQL로 조회할 수 있는 서비스입니다. 프로젝트에서는 과거 데이터 조회를 위해 주로 V1 API를 사용했으며, AVAX, SOL, Ripple 등의 전송 내역을 수집했습니다.

당시 한 번에 최대 25,000건을 조회할 수 있어 `offset` 값을 변경하며 다음 데이터를 요청했습니다. Python의 `requests` 라이브러리로 응답을 받은 뒤 원본을 JSON으로 저장하고, 분석에 사용할 CSV로 변환했습니다.

### Subscan API

AVAIL 데이터 수집에는 Subscan API를 사용했습니다. Bitquery처럼 필요한 전송 내역만 바로 선택하기는 어려웠지만, 블록(block) 단위로 데이터를 수집한 뒤 필요한 이벤트를 추출할 수 있었습니다. AVAIL은 비교적 신생 체인이어서 수집해야 할 블록 수가 많지 않았습니다.

## 검토했지만 제한이 있었던 방법

### Google BigQuery

Solana 전송 데이터의 규모가 커서 Google BigQuery를 이용한 분석을 시도했습니다. Dogecoin, Ripple, Polygon, Solana 등 일부 블록체인 데이터가 공개되어 있어 Python이나 SQL로 조회할 수 있었습니다.

Ripple과 Dogecoin 데이터는 비교적 다루기 쉬웠지만, 주된 대상이었던 Solana는 일부 데이터 조회에도 큰 비용이 예상됐고 로컬 Python 환경에서 처리하기도 어려웠습니다. 이에 따라 Solana 전체 데이터 수집에는 사용하지 않았습니다.

### 체인별 API

각 체인이 제공하는 API도 검토했지만, 대부분 특정 지갑이나 거래를 조회하는 방식이어서 전체 기간 분석에는 적합하지 않았습니다. 일부 API는 블록 전체를 제공해 후처리 가능성을 확인할 수 있었습니다.

- TON: 한 번에 조회할 수 있는 블록 수가 256개로 제한됐고, 블록에 분석과 무관한 정보도 많이 포함되어 있었습니다.
- AVAIL: 체인 API만으로 전체 기간의 전송 내역을 추출하는 방법을 찾지 못했고, 이후 Subscan API를 사용했습니다.
